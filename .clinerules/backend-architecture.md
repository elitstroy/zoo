# 📐 Свод правил и принципов архитектуры

## 1. Структура файлов и папок

```
app/Modules/
└── [ModuleName]/               # Корень модуля (инкапсулирует бизнес-область).
    ├── [ModuleName]ServiceProvider.php # ТОЧКА ИНТЕГРАЦИИ: регистрирует ресурсы и связи в DI-контейнере.
    │
    # ======================= СЛОЙ ПРЕДСТАВЛЕНИЯ (UI / Interfaces) =======================
    # Внешний интерфейс модуля. Обрабатывает ввод, возвращает ответ.
    #
    ├── UI/
    │   ├── Http/               # HTTP-протокол (Web/API).
    │   │   ├── Api/            # API интерфейсы (V1, V2...).
    │   │   │   └── V1/
    │   │   │       ├── Controllers/ # Прием запроса, вызов Application Service.
    │   │   │       ├── Requests/    # Валидация входных данных (FormRequests).
    │   │   │       └── Resources/   # Преобразование данных в формат ответа (JSON).
    │   │   └── Web/            # Контроллеры для традиционного Web-интерфейса.
    │   ├── Console/
    │   │   └── Commands/       # Artisan-команды модуля.
    │   └── Listeners/          # Обработчики событий (очереди, уведомления).
    │
    # ======================= СЛОЙ ПРИЛОЖЕНИЯ (Application Layer) =======================
    # Оркестратор бизнес-сценариев. Реализует конкретные Use Cases.
    #
    ├── Application/
    │   ├── Actions/            # Атомарные бизнес-операции, изменяющие состояние (SRP классы).
    │   ├── Queries/            # Атомарные бизнес-операции, возвращающие данные (SRP классы).
    │   ├── Services/           # Координаторы. Управляют транзакциями и вызывают доменную логику.
    │   └── DTOs/               # Data Transfer Objects. Анемичные объекты для передачи данных.
    │
    # ======================= ДОМЕННЫЙ СЛОЙ (Domain Layer) =======================
    # Ядро бизнеса. Содержит только правила и логику, не зависит от технологий.
    #
    ├── Domain/
    │   ├── Models/             # Eloquent модели (богатые сущности с логикой).
    │   ├── Contracts/          # Интерфейсы репозиториев и сервисов (Abstractions).
    │   ├── Policies/           # Политики доступа (Authorization).
    │   ├── Events/             # События бизнеса (состояние изменилось).
    │   ├── Exceptions/         # Ошибки, специфичные для бизнес-правил.
    │   ├── ValueObjects/       # Сложные типы данных (Money, Weight).
    │   └── Enums/              # Константы состояний и типов.
    │
    # ======================= СЛОЙ ИНФРАСТРУКТУРЫ (Infrastructure Layer) =======================
    # Реализация контрактов. Технические детали: БД, сторонние API.
    #
    ├── Infrastructure/
    │   ├── Persistence/        # Реализация репозиториев (Eloquent/SQL).
    │   ├── Clients/            # Интеграция с внешними сервисами (Stripe, AWS).
    │   └── ExternalServices/   # Адаптеры внешних систем под интерфейсы домена.
    │
    # ======================= КОНФИГУРАЦИЯ И РЕСУРСЫ (Resources) =======================
    # Системные файлы фреймворка. Сгруппированы для отделения от PHP-классов.
    #
    ├── Resources/
    │   ├── config/             # Настройки модуля (order.php).
    │   ├── database/           # migrations, factories, seeders.
    │   ├── routes/             # api.php, web.php.
    │   ├── lang/               # Локализация.
    │   └── views/              # Blade-шаблоны.
    │
    └── Tests/                  # Тесты модуля.
        ├── Feature/            # Интеграционные тесты.
        └── Unit/               # Модульные тесты домена.
```

## 2. Взаимодействие слоёв: кто и за что отвечает

**Фундаментальное правило:** все зависимости направлены *внутрь*, к Домену.

---

### 🔹 Domain Layer (`Domain`)
- **Назначение:** содержит чистую бизнес-логику и правила. Это сердце приложения.
- **Зависимости:** **ни от кого не зависит**.

#### Правила:

- Не содержит ссылок на другие слои (`Application`, `Infrastructure`, `Http`).
- Не знает о базе данных, HTTP-запросах, очередях.
- Содержит:
    - `ValueObjects`
    - `Domain Events`
    - `Domain Services`

#### Допущение: Eloquent внутри домена
Мы сознательно используем **Eloquent Models** как доменные сущности. Это компромисс ради скорости и экосистемы Laravel.

**Разрешённая "магия":**
- `Бизнес-методы сущности.` Методы, изменяющие или проверяющие состояние самой модели.  
```php
$order->markAsPaid();
$promoCode->isValid();
```

- `Проверки и вычисления на своих полях`
```php
$product->isOutOfStock();
$order->totalWithTax();
```

- `Локальные скоупы`
```php
public function scopeActive($query)
{
    return $query->where('is_active', true);
}
```

- `Value Objects и простые типы`
```php
public function totalWithTax(): float
{
    return $this->total + $this->tax;
}
```

**Запрещено:**

- `SQL-запросы, связанные с другими агрегатами`
```php
public function getUsageCount()
{
    return Order::where('promo', $this->name)->count();
}
```

- `Интеграции с внешними сервисами`
```php
public function sendToCrm()
{
    Http::post('crm.api/orders', [...]);
}
```

- `Форматирование данных для UI`
```php
public function getDiscountInfo(): array
{
    return [
        'text' => "Скидка {$this->value}%",
        'color' => 'green',
    ];
}
```

- `Бизнес-операции между несколькими сущностями`
```php
public function applyToOrder(Order $order)
{
    $order->total -= $this->value;
}
```

- `Работа с внешним состоянием`
```
  - Очереди 
  - События системы
  - Email
  - Filesystem
  - Cache / Redis
```

- `Техническая логика фреймворка`
```
(Request, Response, Facades, Jobs, Notifications, Middleware, и т.д.)
```

---

### 🔹 Application Layer (`Application`)

- **Назначение:** оркестрирует доменную логику для выполнения конкретных бизнес-задач (*Use Cases*).
- **Зависимости:** зависит от **`Domain`** и **`Contracts`**.
- **Состав:**
    - **Actions** — атомарные бизнес-операции, **изменяющие состояние** (SRP классы).
    - **Queries** — атомарные бизнес-операции, **возвращающие данные** (SRP классы).
        - Каждый Query класс решает один конкретный сценарий чтения данных.
        - Принимает DTO с фильтрами/параметрами запроса.
        - Возвращает чистые данные или ValueObjects/Results, без форматирования под UI.
    - **Services** — оркестраторы, управляют транзакциями и вызывают доменную логику (Actions и Queries).
    - **DTOs** — Data Transfer Objects для передачи данных между слоями.
    - **Listeners** — обработчики событий, очередей и уведомлений.
---

#### Правила

- Не зависит от `Http` или `Console`.
- Не содержит SQL или логики доступа к данным.
- Работает только с DTO, а не с `Request`.
- Все зависимости внедряются через конструктор в виде **интерфейсов**.

---

#### Разрешённые инфраструктурно-нейтральные утилиты

`Application Layer` может использовать некоторые утилиты Laravel, если они:

- не инициируют IO
- не выполняют SQL
- не строят запросы
- не обращаются к контейнеру

##### Допустимо

- **Illuminate\Support\Collection**  
  Просто структура данных (*in-memory container*).  
  Не выполняет SQL, не делает IO.

- **Carbon**  
  Value Object времени. Используется для работы с датами и интервалами.

- **LengthAwarePaginator**  
  Данные + метаинформация о пагинации (страницы, `total`, `perPage`).  
  SQL уже выполнен на уровне `Infrastructure`.

---

##### Недопустимо

- **Builder**
- **DB**
- **Facades**
- **where()**, **with()**, **load()**
- Любые API, которые инициируют SQL или внешний IO

### 🔹 Infrastructure Layer (`Infrastructure`)
- **Назначение:** реализует контракты и связывает бизнес-логику с технологиями.
- **Зависимости:** зависит от **`Contracts`** и **`Domain`**.

#### Правила:
- Здесь весь "грязный код":
    - Eloquent-запросы
    - вызовы внешних API
    - отправка email
    - слушатели событий
- Пример: `EloquentOrderRepository` внедряется в `Application Layer` через интерфейс.

---

### 🔹 Presentation Layer (`UI`)
- **Назначение:** точки входа в приложение (API, UI, консоль).
- **Зависимости:** зависит от **`Application`**.

#### Правила:
- Максимально "тонкий" слой.
- Только:
    - валидация
    - аутентификация
    - преобразование запроса в DTO
    - вызов `Action`/`Service`
    - формирование ответа (JSON/HTML)

---

## 3. Controller формирует DTO

- **Правило:** Контроллер — единственное место, где `Request` → `DTO`.
- **Почему:** отвязка бизнес-логики от HTTP.

#### Процесс:
1. Валидация через `FormRequest`.
2. Преобразование:
   ```php
   $dto = OrderData::fromRequest($request);
   // или
   $dto = new OrderData(...$request->validated());
   ```
3. В `Application Layer` передаётся только `OrderData`.

---

## 4. Валидация в `FormRequest`

- **Правило:** вся валидация только в `FormRequest`.
- **Плюсы:**
    - контроллеры чище
    - переиспользование
    - рядом с валидацией — `authorize()` для проверки прав

---

## 5. Исключения: кто создаёт и кто обрабатывает

### Кто создаёт (`throw`):
1. **Domain Layer** — бизнес-исключения (часть языка бизнеса):
   ```php
   throw new OrderCannotBeCancelledException(
       'Нельзя отменить уже доставленный заказ'
   );
   ```
2. **Application / Infrastructure** — технические исключения (например, `ProductNotFoundException`).

---

### Кто обрабатывает (`try...catch`):
1. **Application Layer** — никогда. Оно только пробрасывает ошибки.
2. **Presentation Layer** (`Controller`, `Command`) — первый уровень обработки:
   ```php
   try {
       $this->orderService->cancel($order);
   } catch (OrderCannotBeCancelledException $e) {
       return response()->json(['message' => $e->getMessage()], 422);
   }
   ```
3. **Глобальный обработчик** (`App/Exceptions/Handler.php`) — последний рубеж:
    - логирование (Sentry, Flare)
    - красивая страница 500 или JSON-ответ

## 6. Actions и Queries: правила и использование

**Правило:**

- **Actions** — любая операция, которая **изменяет состояние системы**.
- **Queries** — любая операция, которая **возвращает данные** без изменения состояния системы.

**Причина:** соблюдается SRP (Принцип единой ответственности) — каждый класс выполняет **одну конкретную задачу**, будь то изменение состояния или чтение данных.

---

### 🔹 Плюсы такой организации

- **Ясность**: имя класса говорит само за себя (`CreateOrderAction`, `GetProductsQuery`).
- **Тестируемость**: маленький класс легко покрыть юнит-тестами.
- **Простота**: нет "божественных сервисов", где смешано всё подряд.
- **Отделение чтения и записи**: Query и Action чётко разделены, что упрощает поддержку и масштабирование.

---

### 🔹 Терминология: Actions vs Queries vs Services

| Тип | Что делает | Пример | Взаимодействие |
|-----|-----------|--------|----------------|
| **Action** | Изменяет состояние системы | `CreateOrderAction`, `CancelOrderAction` | Вызывается **Service**; **не вызывает Service** |
| **Query** | Возвращает данные | `GetProductsQuery`, `GetOrderDetailsQuery` | Вызывается **Service**, не изменяет состояние |
| **Service** | Оркестратор сценария/воркфлоу, объединяет несколько Actions и/или Queries | `OrderWorkflowService`, `CatalogSearchService` | Вызывает Actions и Queries; может вызывать другие Services, аккуратно, чтобы не создавать циклов |

---

### 🔹 Правила использования

1. **Controller → Service → Actions/Queries**
    - В контроллерах вызываются Services, а не Actions/Queries напрямую (кроме простых read-only Query без сценариев).

2. **Actions**:
    - Выполняют **одну атомарную операцию**, изменяют состояние.
    - **Не вызывают Services**, чтобы избежать нарушения слоёв и циклов зависимостей.

3. **Queries**:
    - Выполняют **один атомарный запрос** к данным.
    - Никогда **не изменяют состояние** и не вызывают Actions/Services.
    - Используют DTO для входных данных и возвращают **чистые данные или Results/ValueObjects**.

4. **Services**:
    - Координируют сценарии, объединяя несколько Actions и/или Queries.
    - Могут вызывать другие Services, но это нужно делать **аккуратно**, чтобы не создавать сложные цепочки и циклы зависимостей.

5. **SRP соблюдается на уровне класса**:
    - Каждый Action и Query отвечает за **одну конкретную задачу**.
    - Services отвечают за **сценарий целиком**, объединяя несколько операций.

## 7. Где можно использовать "магию" Laravel

### Domain Layer — **зона без магии**
✅ Разрешено: Eloquent-модели, Relationships, Mutators, Scopes, Collections.  
❌ Запрещено: фасады, хелперы, Request/Response.  
🕒 **Исключение по времени:** использовать `ClockInterface` (см. раздел Domain/Clock).

### Application Layer — **минимум магии**
✅ Разрешено: DI, `dispatch()`, `event()`.  
⚠️ Фасады — нежелательны.

### Infrastructure Layer — **прагматичная магия**
✅ Разрешено всё для реализации контрактов:  
`Eloquent Query Builder`, `DB::transaction()`, `Cache::remember()`, вызовы API и пр.

### Presentation Layer — **площадка Laravel**
✅ Разрешено всё:
- Route-Model Binding
- Form Requests
- Фасады (`Auth`, `View`, `Redirect`)
- Хелперы (`response()`, `view()`)
- API Resources, Blade

### Shared Kernel — разрешенные зависимости
Чтобы избежать "архитектурного мазохизма", во всех слоях разрешено использование:
- **Carbon/CarbonInterface** — как основной тип для работы с датой/временем.
- **Illuminate\Support\Collection** — как основной тип коллекции (вместо array).
- **LengthAwarePaginator** — как транспорт для пагинированных данных.

*Важно: Эти объекты должны использоваться исключительно как контейнеры данных или Value Objects без вызова IO-операций.*

## 8. Модуляризация (модульный монолит)

**Цель:** держать монолит управляемым и масштабируемым, готовым к выделению микросервисов.

### Принципы
- **Вертикальные срезы:** каждый модуль содержит собственные `Domain`, `Application`, `Infrastructure`, `Presentation`.
- **Общее ядро (Shared):** общие `ValueObjects` и утилиты живут отдельно: `Money`, `Uuid`, `Clock`, `Result`, контракты событий и т.п.
- **Зависимости:**
    - Модули зависят от **Shared** и собственных `Contracts`.
    - Прямые зависимости между модулями избегаются. Если нужно — через **Contracts** другого модуля или **Domain Events**.
- **Границы:** публичный API модуля — только через его `Contracts` или `Http` (роуты). Внутренние классы — `internal` по договорённости (namespace/папка).


### Правила интеграции
- **IoC:** каждое хранилище/шлюз регистрируется в модульном `ServiceProvider`.
- **Миграции/роуты:** лежат внутри модуля и загружаются из него.
- **Тесты:** модульные тесты рядом с модулем; общие — в Shared.
- **Коммуникации:** кросс‑модульные вызовы — через Contracts; асинхронные реакции — через Domain Events.

---

📌 **Итог:**
- `Domain` = чистые правила бизнеса
- `Application` = оркестрация (Services + Actions)
- `Infrastructure` = реализация и интеграции
- `Presentation` = точка входа и адаптация
